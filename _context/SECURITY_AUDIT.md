# NoBC OS — Security Audit

> Read-only audit, 2026-05-29. Scope: tenant isolation, AuthZ/RBAC, payment integrity, webhook signatures, input validation, file upload/SSRF, raw SQL, secrets/PII, rate limiting, dependencies. **No code, schema, or config was modified.** Method: parallel per-class review of every `route.ts` under `app/api/`, every Prisma call site in `app/` + `lib/`, the `nobc-house-phone/` Railway service, and `npm audit`.

## Executive summary

**No Critical findings. The single highest-risk finding is `HIGH-1` — an orphan, RBAC-ungated membership-approval route at `app/api/apply/membership/[id]/approve/route.ts:29`.** It is guarded only by `auth()` + `requireWorkspaceId` (no `requireRole`), so **any signed-in member of a workspace — including a `READ_ONLY` user — can approve/decline/waitlist membership applications, upsert an `APPROVED` Member, and trigger live production email** from `team@thenobadcompany.com`. It is a duplicate/legacy path with no UI caller (the real operator UI hits the STAFF-gated `/api/operator/applications/[id]/approve`), and it evaded the CLAUDE.md "every `/api/operator/* write` is role-gated" guarantee precisely because it lives under `/api/apply/`, not `/api/operator/`. Blast radius is capped to the caller's own workspace (queries are `{ id, workspaceId }`-scoped, so it is **not** cross-tenant), but it defeats the STAFF boundary and is exploitable today. Fix is one line: swap in `requireRole(OperatorRole.STAFF)`, or delete the dead route.

Tenant isolation is otherwise **strong and consistent** — no classic cross-tenant Prisma IDOR was found; `getMemberWorkspaceId`/`requireWorkspaceId` derive workspace server-side from the Clerk org and cannot be influenced by the request body. Stripe signature verification, the authorize→capture flow, refund authorization, the Phase J HMAC (timing-safe both directions), the Twilio inbound HMAC, the `/apply` Zod validation, and all raw-SQL/FTS construction are clean. The next tier down: a forgeable Apple-wallet token (HIGH but inert until `PASSNINJA_*` is set), four HIGH npm-audit advisories (two patchable via `@clerk` bump; the axios SSRF chain is only reachable through the same inert PassNinja SDK), Stripe webhook non-idempotency (double-email/double-Svix on retry), upload MIME-trust, and missing brute-force throttle on the public share-password gate.

## Verified clean (coverage)

**Tenant isolation:** DAM (`asset/[id]`, `/full`, `/thumb`, `assets`, `assets/bulk`, `folders`, `download-zip`, `upload`), share links (operator CRUD + public token routes derive `workspaceId` from the token row), Stripe (`refund`/`capture`/`create-payment-intent`/`checkout`), RSVP/access, House Phone (`/api/sms/*`), MCP tools, agent tools, all `/api/operator/*`, check-in (`/event`+`/walkin` scope by resolved workspace), intelligence, Google wallet (no token bypass). `getMemberWorkspaceId`/`requireWorkspaceId` resolve workspace from the Clerk org server-side — not client-influenceable. Public `/api/apply/membership/[id]` draft routes are capability-URL-by-CUID by design (no session yet, PENDING-only, internal fields stripped).

**Payments / webhooks:** Stripe sig verified via `constructEvent` over the **raw** body; manual capture (`capture_method:'manual'`, re-checks `status==='requires_capture'`, guards double-capture); refund cannot target an arbitrary/over-refund PI (client supplies only `rsvpId`, PI looked up from workspace-scoped row, double-refund 409-guarded). Phase J HMAC-SHA256 **timing-safe both directions** (`crypto.timingSafeEqual`, raw body, length-guarded). Twilio inbound HMAC-SHA1 timing-safe, proxy-aware (`trust proxy`), validated before any DB/AI work. No inbound Svix surface (outbound-only). Share password (scrypt + `timingSafeEqual` + dummy-hash anti-enumeration) and `share_auth` cookie HMAC (timing-safe, HttpOnly, path-scoped, 1h) both correct.

**Input validation:** All `/apply` paths strictly Zod `safeParse` before Prisma (`lib/validation.ts` bounds email/phone/text/answers). No mass-assignment (`...spread` into Prisma is always post-Zod). Non-Zod body routes use manual type guards + allowlists, workspace-scoped.

**Raw SQL:** Only one `$queryRaw` (`app/api/media/dam/assets/route.ts:38`) — fully parameterized; `lib/dam/search.ts` builds every clause with `Prisma.sql` tagged templates incl. the FTS term (`websearch_to_tsquery('simple', ${p.q})` bound, not concatenated); ORDER BY from a fixed allowlist. No `$queryRawUnsafe`/`$executeRawUnsafe` anywhere. `prisma/sql/dam-search-vector.sql` is trigger+GIN only, no user input.

**Secrets / PII:** No hardcoded secret fallbacks (`process.env.X || '...'` → zero source matches). No secrets/stack/raw-Prisma errors returned to clients (domain-error guards return curated messages). Error boundary (`app/error.tsx`) logs with context, renders only static copy. No PII in URL/query strings. Share password/hash, wallet token, auth cookies never logged.

**Rate limiting present:** SMS inbound (10/phone/hour, in-memory, before DB/AI, after sig check); agent route (workspace-scoped).

**Treated as intentional (not flagged):** `/api/sms/*` Clerk-org-membership auth; `/api/check-in/*` `CHECKIN_SECRET` bearer; shared Neon instance with Producer; public `/api/share/token/*` + password cookie; `/api/apply/*` public surface; operator/settings GET reads at `auth()` level; `cron/*` `CRON_SECRET`; `dev/*` `DEV_USER_IDS` allowlist / `NODE_ENV` gate.

## Suggested triage order

1. **HIGH-1** — `apply/membership/[id]/approve` RBAC bypass (live, one-line fix).
2. **HIGH-2** — Apple wallet token signing **before** enabling `PASSNINJA_*`.
3. **HIGH (deps)** — `npm audit fix` for the js-cookie/Clerk chain; decide on the axios/PassNinja chain before go-live.
4. **MEDIUM** — Stripe webhook idempotency; DAM tag fail-closed; share-password brute-force throttle; upload magic-byte check; public `/apply` rate limit.
5. **LOW / Info** — capture role gate, event-hero presign scope, download-cap TOCTOU, `customAnswers` Zod, PII log masking, QR token width, MCP role model (tracked gap).
